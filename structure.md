# 📁 Codebase Structure & Component Map

This document provides a detailed breakdown of the directory layout, codebase design, and key source components for the **SyncZen Local** ecosystem. The project is divided into two primary subsystems: the **Windows Desktop Workstation** (powered by Electron + React) and the **Android Mobile Assistant** (powered by React Native + Expo).

---

## 🏗️ Architectural Overview

SyncZen Local operates on a local-first, server-client architecture within the hotel's local network (LAN):

```
 Hotel-Check-In/ (Workspace Root)
 ├── desktop/            # Desktop Workstation (Electron App + API Server)
 └── mobile/             # Mobile Assistant (Expo React Native Client)
```

1. **`desktop/`**: The host application. It runs the Electron process which manages the application lifecycle and hosts an Express API server. Data is stored in-memory using WebAssembly SQLite (`sql.js`) and periodically synchronized/written to a local `.db` file on disk for persistent storage.
2. **`mobile/`**: The client application. Staff scan the QR code generated by the desktop workstation to couple the device via local WiFi. Check-ins are submitted directly to the desktop API or buffered inside `AsyncStorage` if the network goes offline, with a background sync engine retrying every 15 seconds.

## 🖥️ Desktop Workstation Subsystem (`desktop/`)

The desktop app operates as a dual-role node: a GUI application for the hotel manager and a local LAN server for the mobile clients.

### 1. Main Process (`desktop/src/main/`)
The entry point running inside Node.js. It manages the operating system hooks, Electron window instances, and physical system dependencies:
*   **`index.ts`**: Initial boot manager. Handles application instances, setups hardware acceleration bypass (`--disable-gpu`), builds IPC (Inter-Process Communication) listeners, and executes a shell script on first run to setup inbound Windows Defender Firewall rules for port `8080`.
*   **`db.ts`**: In-memory WebAssembly SQLite database manager (`sql.js`). It creates tables, seeds default rooms, processes queries, and runs database backups. Since the database resides in WebAssembly memory, `db.ts` exports the binary array and writes it physically to `%APPDATA%/SyncZen Local/synczen.db` on every transaction to prevent data loss.
*   **`api.ts`**: An Express server running inside the Electron main process. It handles local networking configurations, resolves LAN IP interfaces, checks incoming Bearer tokens, and registers REST routes for the mobile client.
*   **`config.ts`**: Reads, writes, and rotates the security parameters (pairing token, server port) stored in `config.json`.

### 2. Preload Process (`desktop/src/preload/`)
Operates as a security gateway between the Node.js Main process and the Chrome Renderer process:
*   **`index.ts`**: Uses Electron's `contextBridge` to expose specific IPC commands as functions under `window.api`, allowing the React UI to trigger actions in the main process (like database queries, firewall setups, and token rotations) without exposing raw Node APIs.

### 3. Renderer Process (`desktop/src/renderer/`)
A React 19 single-page application built using Vite:
*   **`App.tsx`**: Standard layouts, sidebars, page routers, and connection health tickers.
*   **`index.css`**: Complete design styling, including dark-mode glassmorphism backgrounds, occupancy status pills, cards, grids, custom gauges, and micro-animations.
*   **`pages/SetupPage.tsx`**: Displays installation steps (powering database, adding firewall rules, generating pairing credentials).
*   **`pages/DashboardPage.tsx`**: Main administrative dashboard showing live metrics, active occupancy counters, and recent registration logs.
*   **`pages/RoomsPage.tsx`**: Handles room inventories, room types, price changes, and maintenance triggers.
*   **`pages/BookingsPage.tsx`**: Displays active guest bookings and lists documents. Clicking a booking card loads guest photos and scanned ID proofs.
*   **`pages/PairPage.tsx`**: Displays connection details and renders a QR code containing `http://<IP>:<PORT>?token=<TOKEN>` for quick mobile setup.

## 📱 Mobile Assistant Subsystem (`mobile/`)

The mobile client is an offline-first Expo application that acts as the registration terminal for front-desk staff.

### 1. Core Lifecycle & Settings (`mobile/App.tsx` & `mobile/src/`)
*   **`App.tsx`**: Sets up navigation containers using React Navigation, routing users between the Pair Screen, Home Screen, and Check-In Wizard.
*   **`src/store.ts`**: The storage interface. Handles configurations (like saving paired server URLs and tokens) and queues offline guest registration payloads inside native `AsyncStorage`.
*   **`src/api.ts`**: Performs HTTP network pings (`/api/health`), handles bearer tokens, and uploads base64 guest profile photos and group ID proofs to the desktop's `/api/photos/upload` endpoint.
*   **`src/sync.ts`**: The synchronization dispatcher. On app startup (and every 15 seconds in the background), it pings the server to verify connectivity. If online, it processes the pending AsyncStorage queue: uploads image assets, resolves their paths, and commits the stay records.

### 2. Mobile User Interfaces (`mobile/src/screens/`)
*   **`PairScreen.tsx`**: Couples the device to the desktop server. It launches the native camera using `expo-camera` to scan the pairing QR code. It also provides text fields for manual network setups.
*   **`HomeScreen.tsx`**: Serves as the launcher panel. Displays the connection status badge (green `Online` or red `Offline`) and shows the count of buffered check-ins awaiting synchronization.
*   **`CheckInWizard.tsx`**: A multi-step guest registration form:
    1.  **Step 1: Party Size Selection** (Selects the number of incoming guests).
    2.  **Step 2: Guest Details** (Collects name, phone, age, sex, and triggers profile photo captures).
    3.  **Step 3: Documents** (Captures a photo of the group's ID proof).
    4.  **Step 4: Room Allocation** (Pulls available rooms from cache, sets stay duration, and writes room notes).
    5.  **Step 5: Review** (Visual confirmation checklist before submission).

### 3. Native Components (`mobile/src/components/`)
*   **`PhotoWidget.tsx`**: An overlay component that opens the native device camera, provides visual capture frames, toggleable flash controls, and returns captured images as compressed base64 URI strings.

## 🔄 Data Synchronization Pipeline

When a check-in is registered on a mobile device, the data traverses the following flow depending on the local network connection state:

```
        ┌───────────────────────────────────────────────────┐
        │  Staff completes Check-In Wizard (mobile device)  │
        └─────────────────────────┬─────────────────────────┘
                                  │
                          Is Server Reachable?
                         /                  \
                     [Yes]                  [No]
                     /                          \
        ┌───────────▼───────────┐      ┌─────────▼───────────┐
        │  Upload image assets  │      │ Buffer images as    │
        │  to /api/photos/upload│      │ base64 strings      │
        └───────────┬───────────┘      └─────────┬───────────┘
                    │                            │
        ┌───────────▼───────────┐      ┌─────────▼───────────┐
        │ Get file disk paths   │      │ Store payload in    │
        │ from server response  │      │ AsyncStorage queue  │
        └───────────┬───────────┘      └─────────┬───────────┘
                    │                            │
        ┌───────────▼───────────┐      ┌─────────▼───────────┐
        │ Submit stay payload   │      │ 15s Background Loop │
        │ to /api/checkin       │◄─────┤ pings for connection│
        └───────────┬───────────┘      └─────────────────────┘
                    │
        ┌───────────▼───────────┐
        │ Express server adds   │
        │ SQLite records        │
        └───────────┬───────────┘
                    │
        ┌───────────▼───────────┐
        │ sql.js db memory is  │
        │ saved to synczen.db   │
        └───────────────────────┘
```
